# Repository Structure

This repository is organized as a monorepo with one dominant Rust workspace (`codex-rs/`) and a small set of root-level JavaScript/TypeScript packages that package or drive the native runtime.

## Root layout

| Path | Purpose |
| --- | --- |
| `README.md` | Public entrypoint for the overall project. |
| `AGENTS.md` | Repository-specific coding instructions for automation agents. |
| `justfile` | Common developer commands; defaults into `codex-rs/`. |
| `package.json` | Root maintenance package for formatting and repo-wide Node tooling. |
| `pnpm-workspace.yaml` | Declares the Node workspace packages. |
| `MODULE.bazel` / `MODULE.bazel.lock` | Bazel module configuration for Rust and release builds. |
| `.github/` | CI, release workflows, issue templates, and repo automation. |
| `docs/` | User and contributor documentation for the CLI and product behavior. |
| `scripts/` | Repo-wide scripts for install, release, and maintenance tasks. |
| `codex-rs/` | Main Rust workspace and runtime implementation. |
| `codex-cli/` | Legacy Node packaging wrapper that launches the native binary. |
| `sdk/typescript/` | TypeScript SDK that shells out to `codex exec`. |
| `shell-tool-mcp/` | Experimental packaged shell-tool assets and tests. |
| `patches/` | Bazel or dependency patches applied at the repository level. |
| `third_party/` | Vendored third-party source/assets used by the repo. |
| `.planning/` | Planning artifacts generated by project workflows. |

## Build and workspace control files

The repo uses three build/workspace systems in parallel:

- Rust workspace root: `codex-rs/Cargo.toml`
- Node workspace root: `pnpm-workspace.yaml`
- Bazel module root: `MODULE.bazel`

`justfile` is the main human-facing entrypoint and assumes `codex-rs/` as the working directory for most commands.

## `codex-rs/` workspace map

`codex-rs/Cargo.toml` declares a wide workspace. The directory names are short, but the crate names are prefixed with `codex-`.

### Frontend and entrypoint crates

These crates expose user-facing binaries or service surfaces:

- `codex-rs/cli/`: multitool `codex` entrypoint
- `codex-rs/tui/`: fullscreen terminal UI
- `codex-rs/exec/`: non-interactive automation entrypoint
- `codex-rs/app-server/`: JSON-RPC server for IDE/editor integrations
- `codex-rs/mcp-server/`: Codex-as-an-MCP-server mode
- `codex-rs/file-search/`: standalone file-search CLI
- `codex-rs/app-server-test-client/`: testing/debug client for app-server
- `codex-rs/debug-client/`: debugging-oriented client tooling
- `codex-rs/responses-api-proxy/`: internal proxy tooling
- `codex-rs/stdio-to-uds/`: stdio-to-Unix-domain-socket relay utility

### Core domain crates

These hold shared product behavior and contracts:

- `codex-rs/core/`: session lifecycle, tools, config loading, skills, memories, MCP, orchestration
- `codex-rs/protocol/`: internal protocol and shared data types
- `codex-rs/app-server-protocol/`: app-server JSON-RPC schema and TS/JSON schema generation
- `codex-rs/state/`: SQLite-backed state/indexing layer
- `codex-rs/config/`: config-related shared types/support crate

### Provider and integration crates

These speak to external APIs or system integrations:

- `codex-rs/codex-api/`: typed provider/API client layer
- `codex-rs/codex-client/`: transport and retry primitives
- `codex-rs/backend-client/`: backend client helpers
- `codex-rs/chatgpt/`: first-party ChatGPT/Codex product integration
- `codex-rs/ollama/` and `codex-rs/lmstudio/`: local model provider adapters
- `codex-rs/rmcp-client/`: MCP client implementation
- `codex-rs/network-proxy/`: local network policy proxy
- `codex-rs/package-manager/`: package-manager integration support
- `codex-rs/artifacts/`: artifact-related functionality
- `codex-rs/cloud-requirements/`, `codex-rs/cloud-tasks/`, `codex-rs/cloud-tasks-client/`: cloud-oriented features

### Security, sandbox, and execution crates

These crates define the execution boundary around commands and file edits:

- `codex-rs/execpolicy/`: current execpolicy engine
- `codex-rs/execpolicy-legacy/`: older execpolicy implementation
- `codex-rs/linux-sandbox/`: Linux sandbox entrypoint/support
- `codex-rs/windows-sandbox-rs/`: Windows sandboxing support
- `codex-rs/process-hardening/`: pre-main hardening
- `codex-rs/shell-escalation/`: shell exec interception/escalation protocol
- `codex-rs/shell-command/`: shell parsing/safety helpers
- `codex-rs/apply-patch/`: patch application support
- `codex-rs/exec/`: automation runner also participates in this boundary

### Support and utility crates

These are smaller focused libraries used across the workspace:

- `codex-rs/ansi-escape/`, `codex-rs/async-utils/`, `codex-rs/arg0/`, `codex-rs/feedback/`
- `codex-rs/hooks/`, `codex-rs/keyring-store/`, `codex-rs/login/`, `codex-rs/otel/`
- `codex-rs/secrets/`, `codex-rs/skills/`, `codex-rs/test-macros/`
- `codex-rs/utils/*`: many narrowly scoped helper crates such as `absolute-path`, `cargo-bin`, `git`, `pty`, `stream-parser`, `approval-presets`, and `sandbox-summary`

## Internal structure of the most important crates

### `codex-rs/core/src/`

This directory is organized by subsystem rather than by technical layer.

High-value subdirectories:

- `agent/`: built-in sub-agent roles and delegation control
- `apps/`: app-related integration helpers
- `auth/`: login/auth abstractions
- `config/`: config types, schema, edits, permissions
- `config_loader/`: config layering and origin tracking
- `context_manager/`: prompt/context assembly
- `instructions/`: system/developer instruction assembly
- `mcp/`: MCP integration logic
- `memories/`: startup memory extraction/consolidation
- `models_manager/`: model catalogs, presets, and collaboration mode configuration
- `plugins/`: plugin/marketplace integration
- `rollout/`: transcript persistence and session indexing
- `sandboxing/`: sandbox policy support
- `skills/`: skill loading and remote-skill support
- `tools/`: tool registry, routing, handlers, runtimes
- `unified_exec/`: unified execution subsystem support

High-value files:

- `codex-rs/core/src/lib.rs`: module graph and public exports
- `codex-rs/core/src/thread_manager.rs`: active-thread factory and registry
- `codex-rs/core/src/codex_thread.rs`: public per-thread conduit
- `codex-rs/core/src/codex.rs`: turn execution engine
- `codex-rs/core/src/client.rs`: provider session/turn client

### `codex-rs/tui/src/`

The TUI is split into feature-oriented UI modules:

- App state and event loop: `app.rs`, `app_event.rs`, `app_event_sender.rs`, `tui/`
- Input and bottom-pane UX: `bottom_pane/`
- Transcript/history rendering: `chatwidget/`, `history_cell.rs`, `render/`, `diff_render.rs`
- Streaming UI behavior: `streaming/`
- Onboarding and setup: `onboarding/`
- Status/account/model UI: `status/`, `status_indicator_widget.rs`, `model_migration.rs`
- Reusable widgets: `public_widgets/`
- Snapshot fixtures: `snapshots/` plus feature-local `snapshots/` folders

### `codex-rs/app-server/src/`

This crate is flatter and more service-oriented:

- Runtime/bootstrap: `lib.rs`, `main.rs`
- Request dispatch: `message_processor.rs`, `codex_message_processor.rs`
- Transport and routing: `transport.rs`, `outgoing_message.rs`
- Thread tracking and status: `thread_state.rs`, `thread_status.rs`
- Focused API helpers: `config_api.rs`, `external_agent_config_api.rs`, `command_exec.rs`, `dynamic_tools.rs`, `models.rs`

### `codex-rs/protocol/src/` and `codex-rs/app-server-protocol/src/`

The contract crates are organized by domain types rather than by transport:

- Internal protocol types: `codex-rs/protocol/src/protocol.rs`, `items.rs`, `approvals.rs`, `models.rs`, `user_input.rs`, `request_user_input.rs`
- App-server wire schema: `codex-rs/app-server-protocol/src/protocol/common.rs`, `v1.rs`, `v2.rs`, `thread_history.rs`
- Schema/export utilities: `codex-rs/app-server-protocol/src/export.rs`, `schema_fixtures.rs`

## Root Node/TypeScript packages

### `codex-cli/`

This is the install wrapper for `@openai/codex`, not the maintained CLI implementation. It mainly contains:

- `codex-cli/bin/codex.js`: platform detection and child-process launch of the native binary
- `codex-cli/scripts/`: packaging/container helpers
- `codex-cli/vendor/` at publish time: bundled native binaries and tools

### `sdk/typescript/`

This is a thin SDK over the native CLI:

- Public exports: `sdk/typescript/src/index.ts`
- Client wrapper: `sdk/typescript/src/codex.ts`
- Thread abstraction: `sdk/typescript/src/thread.ts`
- Process runner: `sdk/typescript/src/exec.ts`
- Type definitions: `events.ts`, `items.ts`, `threadOptions.ts`, `turnOptions.ts`
- End-to-end tests: `sdk/typescript/tests/`

### `shell-tool-mcp/`

This package is a small standalone Node package with packaged assets/tests:

- Runtime/source helpers: `shell-tool-mcp/src/`
- Patched shell sources: `shell-tool-mcp/patches/`
- Jest coverage: `shell-tool-mcp/tests/`

## Generated, vendored, and patched content

Generated or semi-generated locations to recognize:

- `codex-rs/core/config.schema.json`: generated config schema
- `codex-rs/tui/src/snapshots/` and other `*.snap` files: snapshot test artifacts
- `codex-rs/vendor/`: vendored binaries or third-party components used by the Rust workspace
- `third_party/`: repository-level vendored source trees
- `patches/`: Bazel/dependency patch files
- `shell-tool-mcp/patches/`: shell wrapper patches

These directories are not the best place to start when tracing business logic.

## Naming and organization patterns

The repo uses a few consistent patterns:

- Directory names in `codex-rs/` are short, but crate names are `codex-*`.
- Most workspace members are single-purpose crates with one directory per concern.
- Public contracts are split from implementation: `protocol/` and `app-server-protocol/` are separate from `core/` and `app-server/`.
- UI code is feature-oriented; backend/core code is subsystem-oriented.
- Tests are usually colocated with crates, with additional `tests/` directories or `src/*tests.rs` files.
- Snapshot-heavy UI coverage lives directly beside the TUI rendering code.

## Navigation shortcuts

If you are new to the repo, use this order:

1. Root overview: `README.md`
2. Developer commands: `justfile`
3. Native entrypoint: `codex-rs/cli/src/main.rs`
4. Main engine: `codex-rs/core/src/lib.rs`
5. Thread/session orchestration: `codex-rs/core/src/thread_manager.rs`
6. TUI surface: `codex-rs/tui/src/lib.rs`
7. API surface: `codex-rs/app-server/src/lib.rs`
8. SDK wrapper: `sdk/typescript/src/exec.ts`

## Structure summary

In practice, think of the repo as:

- a Rust product workspace in `codex-rs/`
- wrapped by thin install/distribution packages in `codex-cli/` and `sdk/typescript/`
- supported by root-level CI, docs, scripts, Bazel, and pnpm workspace metadata

That mental model makes the directory tree much easier to navigate.
